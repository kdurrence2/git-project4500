Creating a New Repository on GitHub

Before completing this step, it is helpful to have a GitHub account set up. You can set up a GitHub account here.

Once an account has been made, or if you already have an account, you can sign in and continue to the next step.

  1. Create a New Repository
    • On the Home/Dashboard of GitHub, select the “+” symbol on the upper right hand corner and select “New Repository.”
  2. Complete the Form to Finalize the Repository
    • The form will ask you to name your repository, make the repository public (anyone can see) or private (only available to your account). Once it is complete, select “Create Repository.”
  3. A New Repository Has Successfully Been Created!

Cloning a GitHub Repository on VS Code

Cloning a remote repository allows you to create a copy of all code files and directory structure.

  1. Copy the Remote Repository URL
    • You can quickly copy the repository URL by navigating to the “Code” tab, then selecting the green “Code” dropdown and copying the URL from the menu.
  2. Initiate Repository Clone in VS Code
    • Return to VS Code. Select the “+” menu near the “Source Control” header and select “Clone.”
  3. Select “Clone from URL”
    • In the command line near the top of the screen, paste the copied URL to clone the repository from GitHub.
  4. Select the Destination Folder
    • Assign a destination folder for the new repository. This can be a new or existing file.
  5. The Repository Has Successfully Been Cloned!
    • The cloned repository will open in a new VS Code window or you can add it to the current workspace.

Working with Remote Repositories

Remote repositories on GitHub allow you to store your code on the cloud and share this code with potential collaborators. Creating new repositories and switching between projects is important when working on many projects at once.

We have already reviewed how to create a remote repository on GitHub (see here for a reivew). In the next section, we will look at how to creat a remote repository and swap between remote repositories directly from VS Code.

Create a New Remote Repository from VS Code

  1. Open the "Source Control" Menu
    • Select the “View” tab and then select the “Source Control.”
    • Or, select the “Source Control” icon on the Action Bar on the left hand side of the screen.
  2. Select “Open Folder”
    • Select a folder to start or open a local repository on your device.
  3. Select “Publish to GitHub”
    • This will initialize and create a remote repository directly on GitHub. You can start utilizing Git and GitHub version control features.
  4. A New Remote Repository Has Successfully Been Created!

Accessing a Different Remote Repository

You can use this method to access any additional remote repositories on GitHub from VS Code.

  1. Select the Blue “Open Remote Window” Symbol
    • On the bottom right hand corner, select the double arrow “Open Remote Window symbol.
  2. Select “Remote Repository”
    • Select “Remote Repository” from the dropdown list on the command line bar.
  3. Select “Open Repository from GitHub”
    • Select ““Open Repository from GitHub” from the dropdown list on the command line bar.
  4. Select a Remote Repository
    • All available remote repositories on your GitHub account will appear in the drop down menu. Select the appropriate repository to switch over.
  5. You Have Successfully Accessed a Different Remote Repository
    • The selected repository will open on VS Code and you can begin working immediately.

Branching

What is Branching?

Branching is creating copies of code files to work on features or changes without altering the source code files. Bracnhing is important for version control and is very beneficial for feature development and bug fixes

  1. Branching is important for feature development because you can make changes to code without compromising the base branch. Beacuse you create a copy of code files, you ensure that there is always a working code file to fall back on. Additionally, being able to make multiple copies of a source code file allows team members to work on different features at the same time. This increases productivity and keeps developers from overwriting each other's work.
  2. Branching is also important for fixing bugs because, if all new code changes are made in a separate working branch, bugs will be isolated within a specific branch. This means that bugs can be caught before they are introduced into the base branch. Also, branches can be made specifically for bug fixing. These branches would be useful for isolating issues and also keeping a record of any bugs and solutions.

Creating Branches

  1. Generate New Branch
    • Next to the “Source Control” Header, select the menu icon (…). Select “Branch”, then “Create Branch” from the drop down menu.
  2. Name the Branch
    • Type in the name of the new branch in the command line bar.
  3. The New Branch Has Sucessfully Been Created!
    • The new branch's name appear in the bottom left hand corner. This new branch is now the working branch.

Switching Between Branches

  1. Select the Branch Name
    • Select the branch name in the bottom left hand corner. A new menu will dropdown from the command bar.
  2. Select a Different Branch
    • Select the appropriate branch in the command line dropdown list.
    • The new branch's name appear in the bottom left hand corner.

Pull Requests

What are Pull Requests?

  1. Pull requests are a request to compare changes between branches and merge these changes, most commonly upstream into a main/base branch.
  2. Pull requests are an important part of version control because they identify what changes were made, give the opportunity for feedback on changes, and allow multiple contributors to work on a file at one time.
  3. Pull requests open the floor to discuss features, code changes, potential corrections, constructive feedback etc. These discussions can be very helpful for boosting team communication, learning new things, sharing tips, and checking that there are no issues with the code.

Creating Pull Requests

NOTE: Be sure to have two separate branches: a main branch and a working branch. Any code updates should be made within the working branch and later merged into the main branch.

  1. Sync Changes to Git Hub
    • Make changes to a branch, commit these changes, and push these changes to the remote repository on GitHub.
    • NOTE: These are reviews from earlier steps. Click on each concept for a review.
  2. Select “Compare & pull request”
    • Within the yellow banner, select “Compare & pull request” to review branch changes.
  3. Add a Title or Message for the Pull Request
    • Once you have decided to merge the changes to a different branch, you can add a title and description to the pull request.
  4. Review the Changes in the Code
    • These changes will be listed under each commit and color coded based on the appropriate change type.
    • You can also see the code in more detail by clicking on the commit title.
  5. Leave Comments if Needed
    • General comments can be left in the comment box beneath the changes or by hovering over specific lines of code and clicking the blue “+” that appears.
  6. Select "Create Pull Request"
    • Select green "create pull requet button" above the code review section.

Completing a Pull Request

  1. Review the Pull Request Before Merging
    • All the code changes and comments will be accessible in the review screen.
    • You can select each commit title and see the code in detail.
  2. Select “Merge pull request”
    • First, select "Merge pull request" beneath the reviewed information.
    • Next, select "Confirm merge" to finalize the merge.
  3. Your Branches Have Merged Successfully!
    • A success message will appear once the merge is completed. The “New Feature” branch can be closed and deleted.
    • Your merged changes will appear in the main branch.
